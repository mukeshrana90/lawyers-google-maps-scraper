🔴 High priority — money & correctness

1. PPE charges fire on attempted, not successful, enrichment High ✅ Done · 1.0.9

src/main.js

The charge loop pushes email-enrichment, social-enrichment, etc. based only on whether place.website exists — not on whether the enrichment actually returned data. If email extraction returns null, the customer is still billed. This is both a correctness bug and a trust/refund risk on the Apify Store.

Fix: gate each charge on a real result.

if (enrichEmails && place.email) charges.push('email-enrichment');
if (enrichSocials && hasAnySocial(place)) charges.push('social-enrichment');
if (enrichLawyer && place.practiceAreas?.length) charges.push('lawyer-enrichment');

Shipped with two extra guards: feeStructure?.length (an empty array is truthy in JS) and firmSize !== 'unknown' (firm-size classification always runs, so an "unknown" result must not bill).

2. maxResults can be exceeded under concurrency High ✅ Done · 1.0.9

src/main.js (with maxConcurrency: 2)

The counter is checked then incremented non-atomically across concurrent request handlers, and the scope (per-URL vs global) is ambiguous. N search URLs × maxResults can blow past the intended total.

Fix: decide whether maxResults is a global cap. If so, track it on a shared object and stop enqueuing / break once hit. Document the scope clearly in the input schema.

3. Website navigation has no retry High ✅ Done · 1.0.9

src/main.js → page.goto(place.website, …)

Maps navigation uses navigateWithRetry(), but the website visit calls page.goto directly with a 15s timeout. A single timeout hard-fails that branch and can discard an otherwise-good Maps record.

Fix: wrap the website goto in the same retry / partial-load handling, and never let a website failure throw away the Maps data already collected.

🟡 Medium — reliability & performance

4. Anti-detection patterns are too uniform Medium

src/scraper.js, src/main.js

Fixed scrollBy(0, 800) and a hardcoded waitForTimeout(1500) are exactly the uniformity Google flags as bot behavior.

Fix: randomize scroll distance (e.g. 400–1200px) and replace fixed waits with jittered delays via the existing randomDelay() helper.

5. Review-sentiment phase is the biggest cost driver Medium

src/lawyerEnrichment.js

Enabling it adds a third navigation per place (Maps overview → Reviews tab → website), plus ~4500px of scrolling and 10 "See more" clicks.

Fix (confirm first): check whether the initial Maps detail load already exposes enough review text to skip the separate Reviews-tab navigation. If so, cache and reuse it.

6. Brittle hardcoded selectors scattered across files Medium

src/scraper.js, src/lawyerEnrichment.js, src/main.js

Selectors like span.wiI7pd, div[role="feed"], review-tab classes, and consent dialogs are duplicated (consent handling exists in both main.js and scraper.js).

Fix: centralize them in one selectors.js with documented fallbacks. When Google changes a class, it becomes a one-file fix instead of a hunt.

7. Validation gate is too permissive Medium

src/scraper.js

A place with only a name and no phone/website/email is accepted. For a lawyer lead list that record is near-useless.

Fix: require at least one contact method (phone, website, or email) before pushing/charging.

🟢 Lower — robustness & polish

8. Input shape validation Low

src/main.js

A string searchTerms instead of an array silently iterates characters. Coerce/validate input shape at startup and fail loudly on bad input.

9. Metro keyword fallback is loose Low

src/lawyerNiche.js

Substring-matching city names in the address means a suburb 150km out reads as "metro." Prefer the coordinate path; treat keyword matches as low-confidence.

10. Unexplained magic numbers Low

src/main.js, src/scraper.js, src/lawyerNiche.js

Values like staleRounds >= 3, maxConcurrency: 2, and review thresholds 2000 / 500 / 50 should be named constants with a one-line rationale each.

11. No test or lint setup Low

package.json

There is no test/lint tooling today. A few unit tests on the pure-compute functions (classifyFirmSize, email scoring, sentiment) would catch regressions cheaply and run without a browser.

12. Other notes Low

  • Language normalization fallback keeps original casing when a keyword isn't in the normalize map — can emit GERMAN instead of German.
  • applyFirmSizeClassification() both mutates and returns enriched but callers ignore the return — make it consistent.
  • Booking-platform detection relies on substring matching of footer links; it can report Calendly when the live form is something else. Validate against the active DOM.
  • Email regex scans the entire HTML with no size guard — add a length cap for very large pages.
